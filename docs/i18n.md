# Internationalising an Ari Skill

How to add a non-English language to an existing skill, and how the
SDK's `t()` / `get_locale()` / `format_*` helpers fit together.

## Two layers

A localised skill ships **two** parallel sets of files per language:

```
your-skill/
├── SKILL.en.md         ← English manifest: name, description, patterns, body
├── SKILL.it.md         ← Italian manifest: same shape, translated strings
├── strings/
│   ├── en.json         ← English templates for `ari::t(...)` lookups
│   └── it.json         ← Italian templates
└── skill.wasm          (or no wasm for declarative skills)
```

- **`SKILL.{locale}.md`** carries the *fixed* parts of the manifest in
  each locale: localised `name`, `description`, the markdown `body`
  (used as LLM-routing context), and the `metadata.ari.matching.patterns`
  that the keyword scorer matches against. **Structural fields**
  (`id`, `type`, `capabilities`, `behaviour`-shape) **must match
  identically across every locale variant** — the loader rejects the
  install on any mismatch.
- **`strings/{locale}.json`** is a flat key→template map for *dynamic*
  text the skill emits at runtime: `"You have {count} reminders."` and
  similar. Looked up via the SDK's `t()` helper.

`SKILL.en.md` is required (or the legacy bare `SKILL.md`, which the
loader still accepts and treats as English). `strings/en.json` is
required IF the skill ships any other locale's strings; pure-action
skills with no user-facing text don't need a `strings/` dir at all.

The two layers are intentionally separate: a translator can edit
`strings/it.json` without touching the manifest, and a manifest
author can update patterns without touching translations.

## Migrating an existing skill

The legacy `SKILL.md` filename still works, but as soon as you ship
a second language you'll want to rename:

```
git mv SKILL.md SKILL.en.md
```

…and add `SKILL.it.md` alongside. **Hard rule**: a single skill
directory cannot contain both `SKILL.md` and `SKILL.en.md`. The
loader rejects the install. Migration is a rename, not a duplicate.

## SDK helpers

All five are ungated — no capability declaration in `SKILL.md`
needed.

### `ari::get_locale() -> &'static str`

The user's currently-active language as an ISO 639-1 lowercase code
(`"en"`, `"it"`, …). Read fresh on every call from the host's
locale provider.

```rust
let lang = ari::get_locale();
if lang == "it" {
    // dispatch on language-specific behaviour
}
```

### `ari::t(key, args) -> Option<&'static str>`

Look up a translation key in the skill's
`strings/{current_locale}.json` table, falling back to English if
the requested locale doesn't have the key. Substitutes
`{placeholder}` slots from the args slice.

```rust
let greeting = ari::t("greet.hello", &[("name", "Keith")])
    .unwrap_or("Hello!");
// → "Hi Keith!" in English, "Ciao Keith!" in Italian
```

`args` is `&[(&str, &str)]` — name/value string pairs. Numeric
placeholders should be stringified by the caller
(`("count", "3")`); the SDK encodes args as a JSON object before
crossing the WASM boundary.

On a full miss (key absent in both the active locale and English),
the host returns the **bare key** as the resolved string. Visible-
failure UX: a typoed key stays visible to the dev rather than
rendering as empty.

### `ari::format_date(ts_ms, locale, style) -> Option<&'static str>`

Format a Unix epoch millisecond timestamp as a localised date.
Pass `""` for `locale` to use the user's active language. Style is
`"short"`, `"medium"`, `"long"`, or `"full"`; today only `"long"`
is distinct (full month name vs short).

```rust
let when = ari::format_date(reminder_ts_ms, "", "long")
    .unwrap_or("today");
// → "30 April 2026" / "30 aprile 2026" / "30 avril 2026" / …
```

### `ari::format_number(value, locale, style) -> Option<&'static str>`

Format an `f64` using the locale's decimal and group separators.

```rust
let pretty = ari::format_number(1234.56, "", "")
    .unwrap_or("1234.56");
// → "1,234.56" (en) / "1.234,56" (it/de) / "1 234,56" (fr)
```

### `ari::format_currency(amount, currency_code, locale) -> Option<&'static str>`

Format an `f64` with an ISO 4217 currency code in the locale's
customary position. Unknown currency codes pass through verbatim.

```rust
let price = ari::format_currency(1234.56, "EUR", "")
    .unwrap_or("free");
// → "€1,234.56" (en) / "1.234,56 €" (it/fr/es/de)
```

## Patterns of use

**Declarative skill (no code):** the per-locale `response_pick` /
`response` lives in the corresponding `SKILL.{locale}.md`. No SDK
calls needed.

```yaml
# SKILL.it.md
metadata:
  ari:
    declarative:
      response_pick: ["Testa.", "Croce."]
```

**WASM skill (with code):** keep the skill's logic locale-neutral;
all user-facing output goes through `t()` and `format_*`.

```rust
pub fn execute() -> &'static str {
    let count = count_reminders();
    ari::t("reminder.summary", &[("count", &count.to_string())])
        .unwrap_or("You have reminders.")
}
```

`strings/en.json`:
```json
{ "reminder.summary": "You have {count} reminders." }
```

`strings/it.json`:
```json
{ "reminder.summary": "Hai {count} promemoria." }
```

**Skill matching dispatches on locale automatically.** When the
user's locale is `it`, only `SKILL.it.md`'s patterns score against
the input (with a fall-through to English patterns if the skill
ships no Italian variant).

## Supported locales

The host treats locale codes as opaque ISO 639-1 strings, but the
ones we ship resources for today are: **en, it, es, fr, de**.
Adding a new locale is one entry in
`ari_skill_loader::formatters::LOCALES` (engine-side) plus whatever
language data the rest of Ari needs (UI strings, router prompts,
etc. — see the multi-language plan in `wtf.md`).
