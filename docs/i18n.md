# Internationalisation

How to ship a skill in more than one language.

> **The rule that comes first: only ship languages you actually speak.**
> Never machine-translate a skill and never accept a PR that does. A wrong
> translation in a voice assistant is worse than no translation, because the
> user has no way to tell it's wrong. If you want your skill in a language you
> don't have, find a speaker.

---

## Two layers

A localised skill carries two parallel sets of files:

```
your-skill/
├── SKILL.en.md         ← English manifest: name, description, patterns, body
├── SKILL.it.md         ← Italian manifest: same shape, translated
├── strings/
│   ├── en.json         ← runtime text, looked up by key
│   └── it.json
└── skill.wasm
```

**`SKILL.{locale}.md`** carries the parts of the *manifest* that vary by
language: `name`, `description`, the markdown body, and — crucially — the
`matching.patterns`, because Italian users don't say "flip".

**`strings/{locale}.json`** carries the *runtime* text your skill emits:
`"You have {count} reminders."` and friends.

They're deliberately separate. A translator can work on `strings/it.json`
without touching the manifest, and you can retune patterns without touching
translations.

`SKILL.en.md` is required. `strings/` is optional — a skill whose only output
is an action envelope may not need one at all.

### Structural fields must match across locales

`id`, `type`, `capabilities`, and the `declarative`/`wasm` block must be
**identical** in every `SKILL.{locale}.md`. The loader rejects the install if
they diverge. Only the human-facing parts and the patterns vary.

### Migrating from a bare `SKILL.md`

```bash
git mv SKILL.md SKILL.en.md
```

A directory must never contain both `SKILL.md` and `SKILL.en.md` — the loader
rejects it. Migration is a rename, not a copy.

## Declarative skills

The `declarative` block is structural, so it has to be identical in every
locale. You localise by making the response values **keys** rather than text:

```yaml
# identical in SKILL.en.md and SKILL.it.md
    declarative:
      response_pick: ["coinflip.heads", "coinflip.tails"]
```

```json
// strings/en.json                    // strings/it.json
{ "coinflip.heads": "Heads.",         { "coinflip.heads": "Testa.",
  "coinflip.tails": "Tails." }          "coinflip.tails": "Croce." }
```

At execute time each response string is resolved through the table for the
user's language. A value that isn't a key is emitted verbatim, so you can
adopt this gradually.

The validator warns if a response *looks* like a key but isn't in
`strings/en.json` — that's the typo that would otherwise ship as literal
`"coinflip.heads"` in someone's ear.

Patterns, though, are per-locale and should be genuinely different:

```yaml
# SKILL.en.md                         # SKILL.it.md
    matching:                             matching:
      patterns:                             patterns:
        - keywords: [flip, coin]              - keywords: [lancia, moneta]
          weight: 0.95                          weight: 0.95
```

When the user's locale is `it`, only `SKILL.it.md`'s patterns are scored. A
skill with no Italian variant falls through to its English patterns.

**Italian normalisation strips elisions**, so `l'ora` reaches your patterns as
`l ora`. Write `ora`, not `l'ora`.

## WASM skills

Keep the logic locale-neutral and route every user-facing string through
`t()`:

```rust
let count = count_reminders();
ari::t("reminder.summary", &[("count", &count.to_string())])
    .unwrap_or("You have reminders.")
```

```json
// strings/en.json                    // strings/it.json
{ "reminder.summary":                 { "reminder.summary":
    "You have {count} reminders." }       "Hai {count} promemoria." }
```

Do this from day one. Retrofitting `t()` across a finished skill is miserable;
writing it that way from the start costs nothing.

## The SDK helpers

All five are ungated — no capability needed.

### `get_locale() -> &'static str`

The user's active language as a lowercase ISO 639-1 code. Read fresh on every
call.

```rust
if ari::get_locale() == "it" { /* language-specific behaviour */ }
```

Reach for this only when behaviour genuinely differs. If you're using it to
pick which string to return, use `t()` instead.

### `t(key, args) -> Option<&'static str>`

Looks `key` up in `strings/{locale}.json`, falling back to English, and
substitutes `{placeholder}` slots.

```rust
ari::t("greet.hello", &[("name", "Keith")]).unwrap_or("Hello!")
// → "Hi Keith!" (en) / "Ciao Keith!" (it)
```

`args` is `&[(&str, &str)]` — stringify numbers yourself.

On a total miss the host returns the **bare key**. That's deliberate: a typo
stays visible to you instead of rendering as an empty string.

### `format_date(ts_ms, locale, style) -> Option<&'static str>`

Unix epoch milliseconds → a localised date. Pass `""` for `locale` to use the
user's language. Styles: `"short"`, `"medium"`, `"long"`, `"full"` — today
only `"long"` renders differently (full month name).

```rust
ari::format_date(ts_ms, "", "long").unwrap_or("today")
// → "30 April 2026" / "30 aprile 2026"
```

### `format_number(value, locale, style) -> Option<&'static str>`

```rust
ari::format_number(1234.56, "", "").unwrap_or("1234.56")
// → "1,234.56" (en) / "1.234,56" (it)
```

### `format_currency(amount, code, locale) -> Option<&'static str>`

ISO 4217 code, placed where the locale expects it. Unknown codes pass through
verbatim.

```rust
ari::format_currency(1234.56, "EUR", "").unwrap_or("")
// → "€1,234.56" (en) / "1.234,56 €" (it)
```

**Never hand-roll any of these.** Decimal separators, thousands separators and
currency position all vary, and getting them wrong is the sort of thing users
notice immediately.

## Adding a locale to an existing skill

1. `cp SKILL.en.md SKILL.<locale>.md`
2. Translate `name`, `description`, the markdown body — and **rewrite** the
   `matching.patterns` for how people actually speak that language. Do not
   translate the keywords word-for-word; a literal translation of an English
   phrasing is often not what a native speaker would say.
3. Leave `id`, `type`, `capabilities` and the behaviour block untouched.
4. `cp strings/en.json strings/<locale>.json` and translate the values. Keep
   the keys and the `{placeholders}` identical.
5. Add the code to `metadata.ari.languages` in **every** `SKILL.*.md`.
6. Write fresh `examples` in the new language. They're per-locale router
   training data and the same no-poaching rule applies.
7. Validate, then test with the app set to that language.

## Which locales ship today

**English and Italian.** Locale codes are opaque strings to the host, so
nothing stops a third — but the codebase invariant is
`implemented == declared`, so nothing ships a translation it doesn't declare.

Adding a genuinely new language to the *platform* (rather than to one skill)
also needs an entry in `ari_skill_loader::formatters::LOCALES` on the engine
side. Adding a language to a skill, when the platform already supports it,
needs nothing but the files above.

For per-skill translation status, read `index.json` — it's generated from each
skill's frontmatter on publish, so unlike a table in this file it can't go
stale.
